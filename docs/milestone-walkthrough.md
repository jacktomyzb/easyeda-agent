# 里程碑档人工走查 — 脚本与记录(ADR-0002 验收项)

> 交互模式定义:`skills/easyeda-agent/references/design-flow.md` →「交互模式(Interaction Modes)」;
> 决策依据:`skills/easyeda-agent/references/design-decisions.md`;裁决:`docs/adr/0002-*.md`。
> 本文是**走查执行脚本 + 记录表**:用户扮演客户,验证里程碑确认模式的交互体验。每次走查在下方追加一节记录。

## 走查脚本(用户当客户,约 15–30 分钟)

1. **准备**:用户口头/文字给一个板子需求(可以就用 `esp32MiniRequire.md`,或现编一个新需求;**不要**给 BOM/网表/板框等加工过的答案)。
2. **确认点 ①(S0 方案书)**:agent 按里程碑档跑 S0,产出方案书摊给用户。用户重点体验:
   - 决策点是不是**按需求相关性过滤**过(无 USB 的板不该问 USB 取向)?
   - 一次呈现的决策数量是否可消化(目标:一屏内,每条有推荐默认)?
   - 「全部按推荐来」是不是低摩擦一句话就能过?
   - 有没有把 guardrail(唯一正确答案)误摊成选择题?
3. **确认点 ②(转 PCB 前)**:原理图完成后 agent 停住报告。体验:报告是否给了「网表全对/门禁全过」的可判断证据,而不是要用户自己去查。
4. **确认点 ③(交付前)**:P10 双门后 agent 停住。体验:交付摘要是否说清了偏差(如降级决策、遗留 WARN)。
5. **全程反例观察**:中途 agent 有没有在**不该问的地方问**(save/reload/DRC 这类 guardrail),或在**该停的里程碑不停**。
